# 📖 Tech Stack: User Manual

This manual explains how to maintain and evolve the `tech-stack-brain` repository.

## 1. The Purpose of the Tech Stack
This repository is your ultimate "Standard Operating Procedure" for software engineering. If you decide that all future microservices should use `gRPC` instead of a custom `TCP` socket, you document that change **here**. 

Because the central AI Engine reads these folders, any architectural change you make here will instantly apply to all future code generated by the AI across all your connected projects.

## 2. Modifying the Architecture
When introducing a new global rule or pattern:
1. Create a new markdown file in `02-Project-Architecture/`.
2. Add the appropriate Dataview frontmatter (`type: architecture`, `status: active`).
3. Explain the rule clearly using Mermaid diagrams (`.canvas` files) or code blocks.
4. If it is a major decision, write an Architecture Decision Record in `02-Project-Architecture/ADRs/`.

## 3. Modifying Coding Standards
The `03-Project-Coding/` folder acts as the Developer Agent's primary constraint interface.
- If you want the AI to write more robust Rust error handling, update `03-Rust-Safety-and-Async.md`.
- Ensure you provide concrete "Do" and "Do Not" code examples so the AI clearly understands the paradigm.

## 4. Modifying Deployment Scripts
The `04-Project-Deployment/` folder governs the DevOps Agent.
- If you migrate from Docker Compose to Kubernetes, update the files here.
- The AI will read these files to generate `deploy.sh` and CI/CD pipelines.

## 5. Pushing Updates
When you make changes to this repository, you must push them to GitHub. 
Any project that uses this repository as a submodule will need to pull the latest changes:
```bash
git pull origin develop --rebase
```
